# Техническое задание v2 — IrinaOS Мастерская: «Стол и Кабинет»

Заменяет ТЗ v1 («Мастерская», июль 2026). Педагогическое ядро v1 — компасы,
дельта-таблицы, аудит, правило 2–3 подтверждений — сохранено без изменений.
Пересобран хребет: система строится вокруг **текста и его версий**, а не вокруг
учёта диагнозов.

---

## 1. Общая концепция

IrinaOS Мастерская — рабочее место автора, который пишет сам. Приложение
никогда не пишет текст — только диагностирует, спрашивает, приносит справки
и ведёт архив.

У продукта две ипостаси, и они разведены по интерфейсу:

- **Редактор** — работает на **Столе**: смотрит на фрагмент через линзы-компасы,
  ставит диагнозы, задаёт вопросы. Живёт внутри сессии, здесь и сейчас.
- **Секретарь** — работает в **Кабинете**: ведёт Картотеку (корпус), наводит
  Изыскания, составляет Сводки, следит за Портретом голоса, напоминает об
  аудите. Живёт между сессиями, помнит всё.

**Закон итерации (несущий инвариант системы):** каждая итерация заканчивается
правкой текста. Следующий проход по тетради возможен только после фиксации
новой версии фрагмента — той же линзой или другой, но всегда над обновлённым
текстом.

## 2. Словарь системы

Интерфейс и код говорят одним словарём. Технический жаргон
(«раны», «спаунить», «бадж») в UI-слое запрещён.

| Понятие | Термин v2 | Было в v1 |
|---|---|---|
| Главная, место работы Редактора | **Стол** | /workshop (список раундов) |
| Раздел Секретаря | **Кабинет** | — (не существовал) |
| Работа над одним фрагментом | **Тетрадь** | TextThread / цепочка |
| Одна сессия диагностики | **Проход** | PromptRun / раунд |
| Состояние фрагмента в момент времени | **Версия** | — (не существовало) |
| Научная справка | **Изыскания** | research / ресёрч |
| Синтез нескольких проходов | **Сводка** | thread-synthesis |
| Публикация в корпус | **Внести в картотеку** | commit в корпус |
| Убрать из активного, не публикуя | **На полку** | finishedAt / архив |
| Промпт ушёл, ждём ответа | **Депеша отправлена** | status: waiting |
| AUTHOR-VOICE-CORE + дельты + аудиты | **Портрет голоса** | (только md-файлы) |

## 3. Доменная модель

### 3.1 Версия фрагмента (FragmentVersion) — новая сущность

```ts
interface FragmentVersion {
  id: string;
  notebookId: string;
  text: string;              // полный текст фрагмента на этот момент
  createdAt: string;
  note?: string;             // «что изменилось» — формулирует автор при фиксации
  basedOnPassId?: string;    // по итогам какого прохода родилась версия
}
```

Версия — атом системы. Пара соседних версий = пара «исходник ↔ финал» для
шага 3 аудита (ревизия анти-паттернов) и для экстракции механик правки.
Корпус пар растёт сам, как побочный продукт нормальной работы.

### 3.2 Тетрадь (Notebook)

Всё есть тетрадь. Одиночный проход — тетрадь длины один. Отдельной сущности
«standalone раунд» больше нет; отдельного lifecycle для неё — тоже.

```ts
interface Notebook {
  id: string;
  title: string;
  createdAt: string;
  updatedAt: string;
  versionIds: string[];      // FragmentVersion.id, в порядке появления
  passIds: string[];         // Pass.id, в порядке добавления
  committedPath?: string;    // внесена в картотеку
  shelvedAt?: string;        // на полке
}
```

### 3.3 Проход (Pass)

```ts
type PassType =
  | "dry-out"          // Не высушивать: за счёт чего фрагмент живёт
  | "strengthen"       // Усилить: слабые места
  | "mentor-compass"   // Проход по компасу (13 наставников)
  | "inquiry"          // Изыскания
  | "digest"           // Сводка
  | "chekhov";         // legacy, заморожен (clean cutover из v1)

type PassStatus = "draft" | "dispatched" | "completed";
// draft      — черновик, промпт ещё не ушёл
// dispatched — депеша отправлена, ждём ответа
// completed  — диагноз получен и распарсен

interface Pass {
  id: string;
  type: PassType;
  label: string;
  notebookId: string;          // проход всегда живёт в тетради
  fragmentVersionId?: string;  // над какой версией (все типы, кроме inquiry/digest)
  intention?: string;          // «чего вы хотите от этого фрагмента» — уходит в промпт
  inquiryTopic?: string;       // только inquiry
  sourcePassId?: string;       // inquiry: от какого диагноза отправлен секретарь
  compassId?: string;          // mentor-compass
  targetGenreId?: string;      // cross-genre transfer
  promptText: string;          // собранный промпт
  status: PassStatus;
  rawResponse?: string;        // ответ, вставленный автором
  parsedResult?: Record<string, string> | Record<string, string>[];
  completedAt?: string;
  committedPath?: string;
  lastParseFailed?: boolean;
}
```

### 3.4 Закон итерации — как инвариант, а не пожелание

Правила, проверяемые на уровне server actions:

1. Новый проход типа `dry-out` / `strengthen` / `mentor-compass` по тетради
   допустим, только если последний completed-проход этих типов «закрыт»
   версией: существует FragmentVersion с `basedOnPassId` = id этого прохода.
2. `inquiry` — не итерация над текстом: секретаря можно отправить за справкой
   в любой момент, версии не требуется.
3. `digest` работает над тетрадью целиком (≥2 completed-проходов), версии
   не требует и не создаёт.
4. Результаты изысканий **никогда** не подшиваются в следующий промпт
   автоматически. Автор сама решает, брать ли находку в работу.

## 4. Компасы-наставники (13)

Каждый компас = 7 осей диагностики стиля через конкретного автора.
Компас — не имитация: направление — его, походка — ваша.

Реестр: `lib/compasses.ts` — единственный источник правды.

```ts
interface CompassAxis {
  key: string;    // "SORKIN_INTENTION_OBSTACLE"
  label: string;  // "1. Намерение и препятствие"
}

interface CompassMeta {
  id: string;            // "sorkin"
  title: string;         // "Соркин — намерение и препятствие"
  knowledgePath: string; // "learning/compasses/SORKIN-COMPASS.md"
  nativeGenreId: string; // родной жанр
  axes: CompassAxis[];
}
```

Состав:

1. Чехов — сдержанность, подтекст
2. Хемингуэй — айсберг
3. Маркес — бесстрастность к невозможному
4. Зинсер — ясность в нон-фикшне
5. Пратчетт — овеществлённая метафора
6. Толстая — избыточная образность
7. Агата Кристи — честный обман
8. Ле Гуин — мифологическая сдержанность
9. Макфи — архитектура документального текста
10. Манро — переосмысленное время
11. Довлатов — предельная сжатость
12. Уилки Коллинз — скрытая правда через свидетельства
13. **Соркин — намерение и препятствие** *(новый: двигатель сцены,
    драматургия для fiction и нон-фикшна; см. `learning/compasses/SORKIN-COMPASS.md`)*

Механика появления нового компаса, cross-genre transfer
(`nativeGenreId` + подсказки «ПЕРЕНОС» + explicit-абзац в промпте)
и дельта-таблицы — без изменений относительно v1.

### 4.1 Принцип противоречия (новый, явный)

Компасы намеренно противоречат друг другу: Толстая против Довлатова, каскад
Маркеса против отбора Чехова. Система не ведёт к одной эстетике — голос
рождается в выборе между эталонами.

Следствие: **ось может быть переопределена голосом.** Прецедент — аудит
2026-07: избыточность в диалоговой исповедальной прозе признана способом
мышления героини, а не жиром. В дельте такое фиксируется как «уточнена»,
а не «понижена». Машина измерения отступает перед голосом, и это фиксируется
как норма системы, а не как сбой.

## 5. Интерфейс и флоу

### 5.1 Стол (`/desk`) — территория Редактора

Герой экрана — текст. Всегда.

- В центре — **текущая версия фрагмента активной тетради, редактируемая**.
  Это окно — святое; ничто не перекрывает его модалами и не оттесняет.
- Сбоку — лента проходов этой тетради: все свёрнуты, последний раскрыт
  (collapse-by-default сохраняется: диагноз не кричит поверх текста).
- Линзы (Не высушивать / Компас / Усилить) — не формы в отрыве от текста,
  а действия над открытым фрагментом: выбрал линзу — глядя на текст.
- Если активных тетрадей несколько — Стол показывает их стопкой,
  одна всегда раскрыта.

### 5.2 Ритуал итерации

Каждый проход имеет завязку и развязку:

1. **Намерение** — одно поле перед отправкой: «чего вы хотите от этого
   фрагмента?» Уходит в промпт, делает диагностику прицельной.
2. **Депеша** — промпт собран и скопирован; статус «депеша отправлена».
3. **Диагноз** — автор вставляет ответ; парсинг по контракту формата (§6.2).
4. **Правка** — автор работает в окне текста. Редактор молчит.
5. **Фиксация версии** — автор формулирует «что изменилось» (поле `note`)
   и видит **«было → стало»** (diff предыдущей и новой версии). Это пэйофф
   итерации — то, ради чего возвращаются завтра.
6. Дальше: тот же компас над новой версией, другая линза, сводка — или
   тетрадь на полку.

### 5.3 Кабинет (`/study`) — территория Секретаря

- **Картотека** (`/study/card-index`) — корпус: внесённые тетради и проходы,
  `learning/corpus/`.
- **Изыскания** (`/study/inquiries`) — все справки. Источники: вручную;
  от любого диагноза (если в нём есть точка роста); от сводки. Двусторонние
  ссылки тетрадь ↔ изыскание сохраняются.
- **Портрет голоса** (`/study/voice`) — витрина того, что в v1 жило только
  в md-файлах: подтверждённые механики из AUTHOR-VOICE-CORE.md,
  дельта-таблицы всех компасов, история аудитов. Данные остаются в markdown
  (единственный источник правды — файлы), страница их читает и показывает.
- **Полка** (`/study/shelf`) — завершённые тетради. Любую можно вернуть
  на Стол.

### 5.4 Напоминание об аудите

Дисциплина LEARNING-LOOP поручается секретарю: после N зафиксированных
версий с момента последнего аудита (порог в `lib/rituals.ts`, по умолчанию
N = 10) на Столе появляется тихая строка от секретаря: «Накопилось десять
зафиксированных правок — пора сверить портрет голоса». Строка, не модал,
не бадж, не конфетти.

### 5.5 Тон интерфейса

- Редактор прям. Дежурная похвала запрещена; похвала по делу — редка и
  потому драгоценна.
- Секретарь молчалив и точен. Никаких восклицательных знаков в системных
  сообщениях.
- Редактор молчит, пока его не спросили: во время правки (шаг 4 ритуала)
  никаких подсказок, подчёркиваний и «а вы не хотите…».

## 6. BYOK — частная переписка

### 6.1 Принцип (уточнён)

BYOK — это в первую очередь **приватность**, и только потом экономика:
тексты автора не покидают её машину, ни один сервер не логирует ни фрагменты,
ни диагнозы. Приложение никогда не вызывает AI API: собирает промпт →
копирует в буфер → автор вставляет в claude.ai → вставляет ответ обратно.

Ритуал оформлен как характер, а не как неловкость: «передать депешу»,
«депеша отправлена», «ответ получен».

### 6.2 Контракт формата ответа (новое)

Каждый собранный промпт заканчивается инструкцией: ответ — внутри
размеченного блока с именованными секциями:

```
===IRINAOS===
[СЕКЦИЯ: диагноз]
...
[СЕКЦИЯ: точка роста]
...
===КОНЕЦ===
```

Парсер читает только этот блок. При сбое — `lastParseFailed: true`,
raw-ответ сохраняется полностью, автору предлагается поправить разметку
руками (не перезапрашивать).

### 6.3 Prompt Building ≠ Delivery

Без изменений: `lib/prompts.ts` собирает, UI-слой доставляет, слои не знают
друг о друге. Замена ручной доставки на прямой API-вызов — замена одного
компонента.

### 6.4 Два режима доставки — продуктовая позиция (новое)

Ручная доставка — не временный костыль до «настоящего» API, а полноценный
первый режим и часть позиционирования продукта. Тренд на локализацию и
приватность работы с ИИ реален, и для части аудитории «тексты не покидают
вашу машину, вы сами носите депеши в свою ИИ» — решающий аргумент при выборе
и готовый маркетинговый язык. Этот режим закрепляется как носитель ценности
«частная переписка» (§6.1) и не подлежит удалению при появлении API-режима.

Одновременно архитектура обязана допускать лёгкую замену на второй режим —
**личный API-ключ клиента** (если у продукта появятся клиенты-покупатели,
которым ручной ритуал мешает сильнее, чем важен сам жест автономии — это
вопрос исследования аудитории, не предрешаем):

```ts
interface DeliveryMechanism {
  dispatch(pass: Pass): Promise<void>;   // депеша ушла
  receive(pass: Pass): Promise<string>;  // raw-ответ получен
}

// Режим 1 (по умолчанию): ManualDelivery — буфер обмена + ручная вставка
// Режим 2 (future):       ApiKeyDelivery — прямой вызов с личным ключом
//                          клиента; ключ хранится только локально, у нас
//                          его нет — принцип частной переписки сохраняется
```

Переключение режима — настройка, не форк: сборка промптов (§6.3), контракт
формата (§6.2), доменная модель и весь UI Стола не знают, каким способом
депеша дошла до ИИ. Единственное место замены — реализация
`DeliveryMechanism`.

## 7. Архитектурные принципы (СВЯЩЕННЫЕ)

1. **Автор никогда не теряет контроль.** ИИ не пишет ничего — только
   разбирает и предлагает варианты. Изыскания не подшиваются автоматически.
   Тетрадь не продолжается автоматически. Все решения — ручные.
2. **Текст — герой.** Окно фрагмента в центре; закон итерации — инвариант
   модели (§3.4), а не пожелание.
3. **BYOK = частная переписка** (§6). Никакого логирования, полная автономия.
4. **Single Source of Truth.** Компасы — `lib/compasses.ts`; pass types —
   `lib/types.ts` + `lib/passMeta.ts`; знание — markdown-файлы; явные ветки
   в `buildCaseMarkdown`, никаких молчаливых фолбэков.
5. **Полка ≠ Картотека.** `shelvedAt` убирает из активного; `committedPath`
   публикует в корпус. Завершить можно без публикации.
6. **Clean Cutover.** Старый `"chekhov"` путь заморожен и читается; всё новое —
   через `"mentor-compass"`. Миграция v1 → v2 — по §8.2.
7. **Компасы противоречат друг другу намеренно**; голос может переопределить
   ось (§4.1).
8. **Один язык.** Интерфейс говорит словарём системы (§2); два регистра —
   живой в файлах компасов и канцелярский в UI — запрещены.

## 8. Хранение и миграция

### 8.1 Структура `data/`

```
data/
  notebooks.json          # Notebook[]
  passes.json             # Pass[]
  fragment-versions.json  # FragmentVersion[]
  genres.json             # GenreEntry[] (open-data; добавить id "drama" для Соркина)
  chekhov-delta.json      # legacy, не трогается
```

Операции: `readCollection<T>(filename)` / `writeCollection(filename, array)` —
как в v1.

### 8.2 Миграция v1 → v2 (одноразовый скрипт)

- Каждый standalone PromptRun → тетрадь длины один: `sourceText` становится
  версией №1, сам ран — проходом в ней.
- Каждый TextThread → тетрадь: `sourceText` каждого раунда становится
  версией (последовательные дубли схлопываются), раунды — проходами со
  ссылкой на свою версию.
- `finishedAt` → `shelvedAt`; `research` → `inquiry`; `thread-synthesis` →
  `digest`; `waiting` → `dispatched`.
- `"chekhov"`-раны переносятся как есть (тип заморожен).

### 8.3 Корпус

`learning/` (компасы, corpus, audits, AUTHOR-VOICE-CORE.md, EXPERIMENT-LOG.md,
craft-library/) — без изменений; рекомендуется держать под git.

## 11. UI-компоненты (ключевые)

### 11.1 FragmentPane — окно текста (новый, центральный)
- Текущая версия фрагмента, редактируемая; переключатель версий; diff-режим
  «было → стало».
- Во время правки — никаких подсказок (§5.5).

### 11.2 PassCard (бывш. RoundCard)
- Статус («черновик» / «депеша отправлена» / «диагноз получен»),
  промпт свёрнут, кнопка «передать депешу», поле для ответа.
- После ответа: диагноз (свёрнут), точка роста → кнопка «отправить
  секретаря за справкой», raw-ответ.

### 11.3 NotebookCard (бывш. ThreadCard)
- Шапка: название, число версий и проходов, статус.
- Лента проходов (свёрнуты, последний раскрыт), связанные изыскания.
- Активна: линзы + кнопка «сводка» + «внести в картотеку» + «на полку».
- На полке: «вернуть на стол».

### 11.4 Линзы (бывш. три карточки форм)
Три действия над открытым фрагментом: **Не высушивать**, **Проход по
компасу** (селекты: наставник + жанр), **Усилить**. Плюс поле «намерение»
в каждой (§5.2).

## 12. Ограничения и противопоказания

- ❌ НЕ использовать ИИ для написания текста — только диагностика
- ❌ НЕ подшивать изыскания в промпт автоматически
- ❌ НЕ продолжать тетрадь без зафиксированной версии (закон итерации)
- ❌ НЕ смешивать сборку промпта с доставкой
- ❌ НЕ хвалить из вежливости; НЕ показывать модалы поверх окна текста
- ✅ ДА полка без картотеки (`shelvedAt` без `committedPath`)
- ✅ ДА старый chekhov-путь нетронут
- ✅ ДА collapse-by-default для диагнозов
