# Правила проекта Author

Проект: Author (Next.js / TypeScript). Владелец: methodspoint-afk.
Общение — на русском языке.

## Три слоя защиты от потери работы

Эти правила обязательны для каждой задачи разработки. Они существуют, чтобы при
обрыве сессии максимальная потеря была — небольшой кусок недописанного кода с
готовым планом его переделать, а «тихой» потери (когда непонятно, что вообще
делалось) не случалось никогда.

### Слой 1 — отметка «В РАБОТЕ»

В начале каждой задачи разработки, ДО написания кода:

1. Записать в `handoff.md` в раздел «В РАБОТЕ» блок:
   - **Задача** — что делаем, одной-двумя фразами;
   - **План** — шаги, по которым пойдёт работа;
   - **Файлы** — какие файлы будут созданы или изменены;
   - **Критерий готовности** — как поймём, что задача завершена.
2. Сразу закоммитить и запушить этот блок.

По завершении задачи блок «В РАБОТЕ» очищается, а краткий итог переносится в
раздел «Журнал выполненных задач».

### Слой 2 — WIP-ветка

- Незавершённый код пушится в ветку `claude/author-wip`
  (завести её при первой же длинной задаче).
- В ветку `main` переносится только «зелёное» — код, который проверен и
  работает. Рабочая версия проекта не должна страдать от недописанного кода.

### Слой 3 — промпт восстановления

- В конце `handoff.md` лежит промпт восстановления.
- Если сессия оборвалась, пользователь присылает этот промпт в новый чат.
- Получив его, я обязан: сначала показать «что сделано / что потеряно / план
  дальнейших действий», НЕ трогать ветку `main` и НЕ принимать никаких решений
  без явного «ок» пользователя.

## Работа с git

- Коммиты — небольшие, с понятными сообщениями (можно по-русски).
- После каждой завершённой итерации — push в GitHub, чтобы работа не
  оставалась только в контейнере сессии.

## Процедура «main всегда актуален»

`main` — не источник публикации (приложение запускается локально), поэтому
держать его в актуальном виде — чистая подстраховка без риска. Цель: чтобы при
обрыве сессии `main` отставал максимум на одну задачу, а пользователю не нужно
было ни помнить об этом, ни давать команду.

### Основное — автоматически при завершении задачи

Каждый раз, когда задача завершена (зелёные тесты, рабочая ветка запушена,
`handoff.md` обновлён), я в тот же момент, БЕЗ отдельного вопроса пользователю,
переношу это в `main`:

1. открываю PR из рабочей ветки в `main` (или переиспользую открытый);
2. сливаю его кнопкой слияния (merge-commit — запись PR остаётся в истории);
3. перезапускаю рабочую ветку от свежего `main`
   (`git fetch origin main && git checkout -B <ветка> origin/main`), чтобы
   следующая задача шла поверх актуального `main`;
4. пишу пользователю одну короткую строку «main обновлён до …».

В `main` уходит только «зелёное». Пользователь заранее дал постоянное согласие
на этот порядок — переспрашивать не нужно.

### Подстраховка — раз в 3 дня

Триггер-планировщик раз в 3 дня проверяет, не отстал ли `main` от рабочей
ветки (случай, когда сессия оборвалась до слияния). Отстал — догоняю и коротко
пишу. Всё синхронно — молчу, не спамлю.
